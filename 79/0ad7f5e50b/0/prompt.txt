# Plan: Reproduce GRDB SchedulingWatchdog Crash

## Context

A Depop customer reported a crash in their `ATEventsTests` where GRDB's `SchedulingWatchdog.preconditionValidQueue` fails with `EXC_BREAKPOINT`. The root cause: calling synchronous GRDB `DatabasePool` migration methods from a Swift async context. GRDB relies on `DispatchQueue.getSpecific` to verify queue identity, which breaks when the caller is on Swift's cooperative thread pool rather than a traditional GCD queue.

The goal is to create a minimal reproduction project at `~/Downloads/GRDBCrashRepro` that demonstrates the crash and shows the fix.

## Plan

### 1. Create a Swift package at `~/Downloads/GRDBCrashRepro`

**`Package.swift`** - A simple executable + test target with GRDB dependency.

**`Sources/GRDBCrashRepro/Database.swift`** - Minimal reproduction of the pattern:
- A `DatabaseSetup` struct with a `DatabasePool`
- A `setup()` method that creates a `DatabaseMigrator`, registers a simple migration, and calls `migrator.migrate(dbPool)` synchronously
- A static `live` factory that creates an in-memory `DatabasePool`

**`Tests/GRDBCrashReproTests/CrashReproTests.swift`** - Two tests:
- `test_crash_sync_migrate_from_async_context()` - Calls `setup()` from an async test, reproducing the crash
- `test_fix_async_migrate()` - Shows the fix: wrapping the migrate call so it doesn't run directly on the cooperative thread pool (either using GRDB's async write methods or wrapping in a DispatchQueue continuation)
