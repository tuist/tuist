Automations grow an event-driven trigger: pick **Test updated** under **When** and subscribe to the specific changes you care about — `Marked as flaky`, `Unmarked as flaky`, or state transitions to `Enabled` / `Muted` / `Skipped`. Each subscription fires the configured actions the moment the change happens, without waiting for a scheduled evaluation. Common shape: someone manually marks a flaky test → the automation immediately mutes it and pings Slack. Each automation-driven change records a test case event attributed back to the rule that triggered it, so the test case timeline tells you exactly which automation moved the test and why.
