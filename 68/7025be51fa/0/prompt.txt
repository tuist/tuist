# Fix: App targets missing FRAMEWORK_SEARCH_PATHS for transitive dependencies behind dynamic frameworks

## Context

Issue [#9676](https://github.com/tuist/tuist/issues/9676): When using explicit dependencies (the default for `.tuist()` projects), the `ExplicitDependencyGraphMapper` scopes each framework/library target's `BUILT_PRODUCTS_DIR` to a unique subdirectory. It then sets `FRAMEWORK_SEARCH_PATHS` on each framework/library/bundle target to point at all transitive dependencies' scoped directories. However, **app targets are skipped entirely** (early return at line 67), so they never get these search paths.

The previous fix in PR #9681 tried to address this via `searchablePathDependencies` in `GraphTraverser`, but that approach cannot work because:
1. Target-based dependencies become `.product` references which have no `precompiledPath`, so they produce zero `FRAMEWORK_SEARCH_PATHS` entries in `LinkGenerator.setupFrameworkSearchPath`
2. The traversal only looks at direct dynamic framework deps, missing deeper chains

The real fix belongs in `ExplicitDependencyGraphMapper` which already computes the correct `frameworkSearchPaths` for all targets but only applies them to framework/library/bundle types.

## Changes

### 1. `cli/Sources/TuistGenerator/Mappers/ExplicitDependencyGraphMapper.swift`

Restructure `map(_:graphTraverser:debugConfigurations:)` (lines 52-163) so that app targets (and other non-framework consumer targets) also get `FRAMEWORK_SEARCH_PATHS` in their `baseDebug` settings:

- Keep the `frameworkSearchPaths` computation (lines 53-61) unchanged
- Before the early return for non-framework targets (line 63-68), compute and apply `FRAMEWORK_SEARCH_PATHS` (and `ENABLE_TESTING_SEARCH_PATHS` if needed) to ALL target types
- For non-framework/library/bundle targets: apply only these search path settings, then return (no `BUILT_PRODUCTS_DIR` scoping, no copy script)
- For framework/library/bundle targets: continue with existing full logic unchanged

### 2. `cli/Tests/TuistGeneratorTests/ProjectMappers/ExplicitDependencyGraphMapperTests.swift`

- Update the existing `test_map` assertion for the App target (lines 95-103) to expect `FRAMEWORK_SEARCH_PATHS` in `baseDebug` settings pointing to FrameworkA, DynamicLibraryB, and ExternalFrameworkC product dirs
- Add a new test case for the specific bug scenario: `App -> DynamicFramework -> StaticSPMPackage` verifying the App gets search paths for both
