# Build Data Natural IDs: Research Findings

## Xcode / swift-build (SWBBuildService)

### Targets

**Field:** `BuildOperationTargetStarted.guid` (type: `String`)

The `guid` field is documented in the [swift-build source](https://github.com/swiftlang/swift-build/blob/main/Sources/SWBProtocol/BuildOperationMessages.swift) as "an identifier which is unique in a build graph." This corresponds to the PBXNativeTarget GUID from the `.pbxproj` file, a 24-character hex string (96-bit identifier).

**Stability:** These GUIDs are stable across builds of the same project. They are assigned once when the target is created in the Xcode project and persist. However, they are **not stable across project regeneration** -- if Tuist regenerates the project, new GUIDs may be assigned unless the generator is deterministic. Since Tuist controls project generation, it could ensure deterministic GUID generation (hash of project name + target name), which would make them stable.

**Format example:** `2D02E4BD1E0B4A5D006451C7`

**Usability as key:** Good, but needs a deterministic generation guarantee from Tuist. The companion `BuildOperationTargetInfo` provides `name`, `typeName`, `projectInfo`, `configurationName`, and `sdkroot`, which together form a natural composite key: `{project_name}/{target_name}/{configuration}`.

### Tasks

**Field:** `BuildOperationTaskInfo.signature` (type: `BuildOperationTaskSignature`)

The `BuildOperationTaskSignature` is an enum with three variants, each wrapping a `ByteString`:
- `taskIdentifier(ByteString)` -- prefixed with byte `0`
- `activitySignature(ByteString)` -- prefixed with byte `1`
- `subtaskSignature(ByteString)` -- prefixed with byte `2`

The signature for a task like `CompileSwift` is a human-readable string in the format:

```
CompileSwift normal x86_64 /Users/.../Sources/MyFile.swift
```

This comes from the `ruleInfo` field on `BuildOperationTaskInfo`. The underlying llbuild system uses these as rule keys, and they are [documented as stable](https://gist.github.com/lalunamel/716de8bb16cbf1d942324fc2120931ee): "Every Command has a unique identifier which can be used to refer to the command, and which is expected to be stable even as the build graph is updated by the generating tool."

**Stability:** The signature is stable across builds when the same file is compiled with the same architecture and build configuration. It changes when the file path changes or the architecture changes. The underlying content is a content-addressed hash of inputs (stat info, compiler version, etc.) for cache invalidation, but the rule info string itself is the stable identifier.

**Format examples:**
- `CompileSwift normal arm64 /path/to/File.swift`
- `CompileC /path/to/output.o /path/to/File.m normal arm64 objective-c`
- `Ld /path/to/output normal arm64`
- `CodeSign /path/to/App.app`

**Usability as key:** Excellent. You can SHA-256 hash the signature string to get a fixed-length key. The signature is deterministic for the same task across builds. The `ruleInfo` string is human-readable and makes a good display name.

### Diagnostics

**Field:** `BuildOperationDiagnosticEmitted` has no unique ID of its own.

It has a `LocationContext` enum:
- `.task(taskID: Int, taskSignature: BuildOperationTaskSignature, targetID: Int)` -- linked to a specific task
- `.target(targetID: Int)` -- linked to a target
- `.globalTask(taskID: Int, taskSignature: BuildOperationTaskSignature)` -- build-wide
- `.global` -- no attachment

The `taskID` and `targetID` are sequential integers assigned per build session and are **not stable** across builds.

**Usability as key:** You would need to generate a synthetic ID. A good approach: `SHA256(taskSignature + ":" + file + ":" + line + ":" + column + ":" + message_hash)`. The task signature provides the stable anchor; file/line/message provide uniqueness within a task.

## Bazel (BEP)

### Targets

**Identifier:** `BuildEventId.TargetCompletedId` contains:
- `label` (string): e.g., `//src/main:app`
- `configuration` (`ConfigurationId`): build configuration hash
- `aspect` (string): optional aspect identifier

Source: [build_event_stream.proto](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/buildeventstream/proto/build_event_stream.proto)

**Stability:** Bazel labels are fully stable -- they are the canonical way to refer to targets in the BUILD file. The `configuration` hash changes only when build settings change (e.g., switching from debug to release, or changing platform). The label alone is stable across builds with the same configuration.

**Usability as key:** Excellent. `//src/main:app` is already a natural, human-readable, stable key. For cross-configuration comparison, use `label + configuration_hash`.

### Actions

**Identifier:** `BuildEventId.ActionCompletedId` contains:
- `primary_output` (string): the output artifact path
- `label` (string): owning target label
- `configuration` (`ConfigurationId`): configuration hash

The `ActionExecuted` payload adds: `type` (action mnemonic like `CppCompile`, `Javac`, `SwiftCompile`), `success`, `exit_code`, `start_time`, `end_time`, `command_line`.

**Stability:** The `primary_output` path is deterministic for the same target/configuration/source file. The combination of `label + primary_output` is stable across builds. Note that by default, only failed actions emit `ActionExecuted` events; you need `--build_event_publish_all_actions` to get all of them.

**Usability as key:** Good. Use `SHA256(label + ":" + primary_output)` or just the primary_output path (already unique within a build).

### Tests

**Identifier:** `BuildEventId.TestResultId` contains:
- `label` (string): test target label
- `configuration` (`ConfigurationId`)
- `run` (int32): test run index (for `--runs_per_test`)
- `shard` (int32): shard index (for `--test_sharding_strategy`)
- `attempt` (int32): retry attempt number

**Stability:** The label is fully stable. `run`, `shard`, and `attempt` are deterministic for the same test configuration. The composite key `label + run + shard` identifies a unique test execution slot, while `attempt` differentiates retries.

**Usability as key:** Excellent. `{label}/{run}/{shard}` is a natural, stable key per test result.

## Gradle

### Tasks

**Identifier:** Task path, e.g., `:app:compileKotlin`

The [Gradle Task API](https://docs.gradle.org/current/javadoc/org/gradle/api/Task.html) defines the path as `{project_path}:{task_name}`, where the project path is the fully qualified path in the multi-project hierarchy.

**Stability:** Completely stable across builds. The task path is defined by the build script structure and does not change unless the build configuration changes. Gradle Build Scans use task paths as the primary identifier for cross-build comparison.

**Usability as key:** Excellent. `:app:compileKotlin` is already a natural, human-readable, stable key.

### Dependencies

**Identifier:** GAV coordinates: `group:artifact:version`

**Stability:** Fully stable for resolved dependencies. The resolved version is deterministic when using a lock file. Without a lock file, dynamic versions (e.g., `1.+`) may resolve differently across builds.

**Usability as key:** Good. `com.google.guava:guava:32.1.3-jre` is already a natural key. For the dependency graph, you might also include the configuration name: `{configuration}/{group}:{artifact}:{version}`.

### Tests

**Identifier:** `{class_name}#{method_name}` from JUnit XML output. Gradle writes results to `build/test-results/test/TEST-{class_name}.xml`.

**Stability:** Fully stable. Class and method names are determined by the source code and do not change across builds unless the test code changes.

**Usability as key:** Excellent. `com.example.MyTest#testSomething` is a natural, stable key.

## Answering Your Questions

### 1. Can the IDs be derived naturally from the build data?

**Yes, with caveats:**

| Build System | Entity | Natural ID | Stability |
|---|---|---|---|
| Xcode | Target | `guid` or `{project}/{target}/{config}` | Stable if project is not regenerated; composite key always stable |
| Xcode | Task | `signature` / `ruleInfo` string | Stable across builds (same inputs) |
| Xcode | Diagnostic | None natural | Must synthesize: `hash(task_signature + file + line + message)` |
| Bazel | Target | `label` | Always stable |
| Bazel | Action | `label + primary_output` | Stable (same config) |
| Bazel | Test | `label + run + shard` | Stable |
| Gradle | Task | Task path (`:app:compileKotlin`) | Always stable |
| Gradle | Dependency | GAV (`group:artifact:version`) | Stable with lock files |
| Gradle | Test | `class#method` | Always stable |

### 2. Are they stable enough across builds for comparison?

**Yes.** All three systems have identifiers designed for incremental build comparison. The weakest link is Xcode diagnostics (no natural ID) and Xcode target GUIDs when projects are regenerated. The recommendation is:

- For Xcode targets, use a composite key: `{project_name}/{target_name}/{configuration_name}` rather than the raw GUID.
- For Xcode tasks, use the `ruleInfo` / signature string (or its SHA-256 hash).
- For Xcode diagnostics, synthesize from `task_signature + location + message`.

### 3. What would the cardinality be?

| Build System | Targets | Tasks per target | Diagnostics | Total objects per build |
|---|---|---|---|---|
| Xcode (medium app) | 10-50 | 50-500 | 0-100 | 500-25,000 |
| Xcode (large app, 500+ modules) | 200-1000 | 100-1000 | 0-500 | 20,000-1,000,000 |
| Bazel (large monorepo) | 1,000-50,000 | 1-10 per target | N/A (in action logs) | 1,000-500,000 |
| Gradle (Android app) | 5-20 | 20-100 | 0-50 | 100-2,000 |

### 4. Object storage performance at scale?

**One file per item would be problematic at scale.** For a large Xcode project with 1M tasks across many builds:

- S3 handles 3,500 PUT/s and 5,500 GET/s per prefix. With prefix partitioning by `{project_id}/{report_id}/`, you get independent rate limits per build report, which is fine.
- The real cost is **listing operations and per-request overhead**. Thousands of 1KB gzipped JSON files waste S3 API calls (each GET has ~50ms latency + per-request cost). Reading 1000 tasks would be 1000 GET requests.
- [AWS recommends compacting small files](https://aws.amazon.com/blogs/storage/optimizing-storage-costs-and-query-performance-by-compacting-small-objects/) for query performance.

**The grouped-by-target alternative is much better:**

```
build-reports/{project_id}/{report_id}/targets.json.gz        # index: ~10KB
build-reports/{project_id}/{report_id}/targets/{target_id}/tasks.json.gz       # ~5-50KB
build-reports/{project_id}/{report_id}/targets/{target_id}/diagnostics.json.gz # ~1-10KB
```

For a 50-target project, this is ~100 objects instead of ~5,000. For a 500-target project, ~1,000 objects instead of ~500,000. Each file is a meaningful chunk (5-50KB gzipped) rather than a tiny fragment.

**Recommended approach:**

```
build-reports/{project_id}/{report_id}/summary.json.gz
    # Top-level: build duration, result, environment, target count
    
build-reports/{project_id}/{report_id}/targets.json.gz
    # Array of: {id, name, duration, task_count, error_count, warning_count}
    # An MCP tool or CLI can fetch this single file to show a build overview
    
build-reports/{project_id}/{report_id}/targets/{target_id}.json.gz
    # All tasks + diagnostics for one target
    # {target_info, tasks: [...], diagnostics: [...]}
    # An MCP tool fetches one target at a time for drill-down
```

For the `target_id` in the path:
- **Xcode:** `SHA256("{project_name}/{target_name}")` truncated to 16 hex chars -- stable, URL-safe, collision-free at this scale
- **Bazel:** URL-encode the label, or `SHA256(label)` truncated
- **Gradle:** URL-encode the task path, or `SHA256(task_path)` truncated

This gives you 2-3 objects per target (or just 1 combined file), with the `targets.json.gz` index enabling fast overview without loading all detail. A typical build would produce **1 summary + 1 index + N target files** where N is 10-1000, each file being 5-100KB gzipped -- well within S3 best practices.</result>
