# Kura Controller

This module contains the Kubernetes controller that reconciles Kura account endpoint custom resources.

## Scope

- API group: `kura.tuist.dev`
- Primary resource: `KuraInstance`
- Controller output: Kubernetes workload resources for one account-region Kura deployment.

## Deployment Topology

- **Cache traffic (HTTPS)**: each `KuraInstance` with `spec.publicHost` set gets a provider-specific cloud LoadBalancer in TCP-passthrough mode targeting the runtime's TLS-terminating HTTPS port (`4443`). The controller selects annotations from `--load-balancer-provider` (`hetzner` by default, or `vultr`). When the controller's `--grpc-cluster-issuer` flag is set, the controller creates a cert-manager `Certificate` per instance against that ClusterIssuer; the issued Secret is mounted into the runtime, which terminates TLS itself. Health checks use TCP against the passthrough Service port. Provider-managed certificates are not used because Tuist's `kura.tuist.dev` zone lives on Cloudflare, so cloud providers cannot validate the domain they would issue for.
- **gRPC for Bazel**: setting `spec.grpcPublicHost` provisions a separate provider-specific LoadBalancer in TCP passthrough mode. When `--grpc-cluster-issuer` is set, the controller creates a cert-manager `Certificate` per instance; the issued Secret is mounted into the runtime, which terminates TLS itself. With no issuer the gRPC port is exposed plaintext — only acceptable for kind/dev clusters.
- **Single-AZ regions**: each managed region is currently one cloud provider location without zones. Topology spread is hostname-only — three replicas tolerate one node loss, but a location-wide outage takes the whole regional Kura down. Cross-region failover is the user's responsibility (provision Kura in multiple regions; CLI/Bazel can target any of them).
- **Primary-pod routing**: the public HTTPS and gRPC Services select a single pod via `statefulset.kubernetes.io/pod-name`, not the whole StatefulSet. The mesh replicates writes between pods asynchronously, so fanning the public LoadBalancer across all replicas means a read can land on a pod that has not yet received a just-written blob — inconsistent reads in steady state. Pinning every region's traffic to one primary makes steady-state reads read-your-writes consistent without touching the pure-async write path. Selection is sticky: the controller keeps the current primary (read back from the Service selector) while it stays runtime-routable: Kubernetes Ready, `/status/rollout` serving, writer lock owned, and enough ring visibility for the replica count. If the primary is Ready but runtime-isolated, the controller fails over to the lowest-ordinal runtime-routable pod. If runtime probes are unavailable, the controller falls back to Kubernetes Ready so reconciliation does not flap on transient probe failures. A pod watch re-runs selection on readiness changes. The headless Service keeps the broad selector so peer replication still reaches every pod. This trades read scaling (one pod serves a region's reads) for consistency; deploys stay zero-downtime because the role hands off only to a pod that can see the mesh.
- **Peer replication mTLS**: by default each `KuraInstance` gets a controller-owned `<instance>-peer-tls` Secret with a private per-instance CA and a peer certificate valid for the StatefulSet pod DNS names. The runtime mounts that Secret and uses `https://...:7443` plus `KURA_INTERNAL_TLS_*` for artifact replication and internal status traffic. When `spec.peerTLSSecretName` is set, the controller mounts that externally managed Secret instead and does not generate per-instance material; use this only after the shared cross-region peer CA/cert Secret exists in every target regional cluster.
- **Cross-region peer gateway**: `spec.peerPublicHost` plus `spec.globalDiscoveryDNSName` creates a dedicated peer LoadBalancer Service pinned to the same selected primary pod as the public Services. Kura's global discovery env is only enabled when `spec.peerTLSSecretName` is also set, so the peer gateway plumbing can reconcile before the shared cross-region peer TLS Secret is rolled out.
- **Per-tenant isolation**: a `NetworkPolicy` per `KuraInstance` restricts ingress to (a) pods of the same instance and (b) cluster-namespaced peers reaching `http`/`grpc` only. Egress is unrestricted because the runtime needs to reach the Tuist API and the OTLP collector.
- **Telemetry defaults**: controller-managed Kura pods always get the Prometheus scrape annotations expected by `infra/helm/k8s-monitoring`, the controller injects the cluster deployment environment through `KURA_OTEL_DEPLOYMENT_ENVIRONMENT`, and when the controller's `--otlp-traces-endpoint` flag is set they also get a default `KURA_OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`. Kura uses the same deployment environment for Sentry when `KURA_SENTRY_DSN` is present in `kura-shared-secrets`. `spec.extraEnv` stays authoritative for per-instance overrides.
- **PVC retention**: the StatefulSet uses `whenDeleted: Delete` so a destroyed `KuraInstance` releases its hcloud volumes, and `whenScaled: Retain` so a replica scaling event keeps its cache.
- **Resource limits**: the controller deliberately does **not** set `KURA_FILE_DESCRIPTOR_POOL_SIZE`, `KURA_MEMORY_SOFT_LIMIT_BYTES`, etc. The runtime derives those at startup from the pod's cgroup memory limit and `RLIMIT_NOFILE`. Tune the pod-level `resources.limits.memory` instead — the soft/hard application limits are 70% / 85% of that.

## Development

- Run tests with `go test ./...` from this directory.
- Keep generated CRDs in `infra/helm/tuist/crds/` aligned with API changes.
- Keep the controller independent from the Scaleway Apple Silicon CAPI provider. Kura endpoint lifecycle is a product workload concern, not a macOS node infrastructure concern.
- cert-manager must be installed in the cluster before turning on `--grpc-cluster-issuer`. The controller does not provision the issuer itself. The same issuer is used for both the public HTTPS and gRPC Certificates.
