# Kura Controller

This module contains the Kubernetes controller that reconciles Kura account endpoint custom resources.

## Scope

- API group: `kura.tuist.dev`
- Primary resources: `KuraInstance` and `KuraGateway`
- Controller output: Kubernetes workload resources for one account-region Kura deployment, plus optional dedicated ingress-nginx gateway infrastructure for accounts that should not share the regional gateway.

## Deployment Topology

- **Cache traffic (HTTPS)**: each `KuraInstance` with `spec.publicHost` set gets an nginx Ingress on `spec.ingressClassName` (managed regions default to dedicated shared regional Kura ingress controllers). The controller creates a cert-manager `Certificate` per public host against `--grpc-cluster-issuer`; ingress-nginx terminates TLS and streams uploads/downloads to the instance's internal ClusterIP Service on the runtime's plain HTTP port (`4000`). The backend Service is not a Hetzner `LoadBalancer`.
- **Dedicated customer gateways**: server policy can emit a `KuraGateway` with an opaque gateway ID for enterprise/high-volume accounts. The controller reconciles that CR into a dedicated ingress-nginx `Deployment`, `LoadBalancer` `Service`, ConfigMap, and `IngressClass`; the corresponding `KuraInstance` points at that class. Do not add customer-specific ingress-nginx aliases or customer names to Helm values.
- **gRPC for Bazel (single host)**: gRPC (Bazel REAPI) co-hosts on `spec.publicHost` — there is no separate `grpc.<host>` hostname. Any non-private `KuraInstance` with `spec.publicHost` set gets a second nginx Ingress (same Kura ingress class) on `spec.publicHost`, path-routed to the gRPC service prefixes (`/build\.bazel\.remote\.execution\.v2\.`, `/google\.bytestream\.`) via the `use-regex` annotation, routing to the internal Service's `grpc` port (`50051`) with `backend-protocol: GRPC`. The Service port number is a stable dial-address compat surface; where it lands inside the pod is version-gated per instance: for co-hosted images (releases after `kura@0.10.15`, plus any non-semver tag such as the per-commit `sha-<…>` staging builds) the pod's named `grpc` containerPort points at the co-hosted HTTP port (`4000`, where the runtime serves REAPI h2c alongside the cache API) and `KURA_GRPC_PORT` is not set; pre-cohosted images keep the dedicated `50051` listener wiring. The gate (`cohostedGRPC`, keyed on `spec.image` against `lastDedicatedGRPCListenerVersion`) flips ports and image in the same StatefulSet template update, so mid-roll each pod's named port resolves to wherever that pod's image actually serves gRPC. `spec.grpcPublicHost` is retained only for backward compatibility and is ignored. ingress-nginx applies `backend-protocol` per location when two Ingresses share a host, so the gRPC paths use `grpc_pass` while the REST cache keeps `proxy_pass` on `/`. This second Ingress declares no TLS of its own; the public `Certificate`/Secret covers the shared host. No dedicated gRPC `Certificate` is provisioned (the old `<name>-grpc-tls` cert/Secret are deleted on reconcile). Regex paths are required: a plain `Prefix` path renders as `location /build.bazel.remote.execution.v2./`, which a real method path (`…v2.Capabilities/GetCapabilities`) would not match. The path values start with `/` (the Ingress API rejects a path that does not); ingress-nginx prepends the `^` anchor itself under `use-regex`, rendering `location ~* "^/build\.bazel\.remote\.execution\.v2\."`. **Hard dependency on ingress-nginx:** the path-split relies on the `nginx.ingress.kubernetes.io/*` annotations (`use-regex`, per-location `backend-protocol`), which only ingress-nginx honors — so the IngressClass the Kura Ingress targets must be served by ingress-nginx (Tuist's shared regional Kura ingress controllers and the dedicated `KuraGateway` both are). On a non-nginx controller (Traefik/Contour/HAProxy/Cilium) those annotations are ignored and gRPC silently routes to the HTTP backend via `proxy_pass`. The controller cannot enforce this — it emits the annotations regardless of which controller backs the class — so single-host tightens the ingress-nginx assumption that split-host could have side-stepped per gRPC host.
- **Single-AZ Hetzner**: each managed product region is one Hetzner location (`fsn1`/`ash`/`hil`) represented by a dedicated node pool in the main production workload cluster. Topology spread is hostname-only — three replicas tolerate one node loss, but a location-wide outage takes the regional Kura endpoint down. Cross-region failover is the user's responsibility (provision Kura in multiple regions; CLI/Bazel can target any of them).
- **Primary-pod routing**: the public HTTP/gRPC backend Service selects a single pod via `statefulset.kubernetes.io/pod-name`, not the whole StatefulSet. The mesh replicates writes between pods asynchronously, so fanning a public endpoint across all replicas means a read can land on a pod that has not yet received a just-written blob. Pinning every region's traffic to one primary makes steady-state reads read-your-writes consistent without touching the pure-async write path. Selection is sticky: the controller keeps the current primary (read back from the Service selector) while it stays runtime-routable: Kubernetes Ready, `/status/rollout` serving, writer lock owned, and enough ring visibility for the replica count. If the primary is Ready but runtime-isolated, the controller fails over to the lowest-ordinal runtime-routable pod. If runtime probes are unavailable, the controller falls back to Kubernetes Ready so reconciliation does not flap on transient probe failures. A pod watch re-runs selection on readiness changes. The headless Service keeps the broad selector so peer replication still reaches every pod. This trades read scaling (one pod serves a region's reads) for consistency; deploys stay zero-downtime because the role hands off only to a pod that can see the mesh.
- **Peer replication mTLS**: by default each `KuraInstance` gets a controller-owned `<instance>-peer-tls` Secret with a private per-instance CA and a peer certificate valid for that StatefulSet's pod DNS names. The runtime mounts that Secret and uses `https://...:7443` plus `KURA_INTERNAL_TLS_*` for artifact replication and internal status traffic.
- **Cross-region peer discovery**: the controller still supports an explicit `spec.peerTLSSecretName` escape hatch that mounts externally managed shared peer material and injects account-scoped discovery (`KURA_GLOBAL_DISCOVERY_DNS_NAME`). Do not enable it by default until the shared certificate SAN model matches the DNS name Kura advertises in `KURA_NODE_URL`. There is no public peer gateway or per-customer peer `LoadBalancer`.
- **Per-tenant isolation**: a `NetworkPolicy` per `KuraInstance` restricts ingress to (a) pods of the same instance, (b) same-account Kura peers on the mTLS peer port, and (c) cluster-namespaced peers reaching `http`/`grpc` only. Egress is unrestricted because the runtime needs to reach the Tuist API and the OTLP collector.
- **Telemetry defaults**: controller-managed Kura pods always get the Prometheus scrape annotations expected by `infra/helm/k8s-monitoring`, the controller injects the cluster deployment environment through `KURA_OTEL_DEPLOYMENT_ENVIRONMENT`, and when the controller's `--otlp-traces-endpoint` flag is set they also get a default `KURA_OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`. Kura uses the same deployment environment for Sentry when `KURA_SENTRY_DSN` is present in `kura-shared-secrets`. `spec.extraEnv` stays authoritative for per-instance overrides.
- **PVC retention**: the StatefulSet uses `whenDeleted: Delete` so a destroyed `KuraInstance` releases its hcloud volumes, and `whenScaled: Retain` so a replica scaling event keeps its cache.
- **Resource limits**: the controller deliberately does **not** set `KURA_FILE_DESCRIPTOR_POOL_SIZE`, `KURA_MEMORY_SOFT_LIMIT_BYTES`, etc. The runtime derives those at startup from the pod's cgroup memory limit and `RLIMIT_NOFILE`. Tune the pod-level `resources.limits.memory` instead — the soft/hard application limits are 70% / 85% of that.

## Development

- Run tests with `go test ./...` from this directory.
- Keep generated CRDs in `infra/helm/tuist/crds/` aligned with API changes.
- Keep the controller independent from the Scaleway Apple Silicon CAPI provider. Kura endpoint lifecycle is a product workload concern, not a macOS node infrastructure concern.
- cert-manager must be installed in the cluster before turning on `--grpc-cluster-issuer`. The controller does not provision the issuer itself. The flag name is historical: it now issues only the single public-host `Certificate` (gRPC co-hosts on it; there is no separate gRPC certificate).
