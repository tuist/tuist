# Сообщение об ошибках {#issue-reporting}

Как пользователь Tuist, вы можете столкнуться с ошибками или неожиданными
действиями. В этом случае мы рекомендуем вам сообщать о них, чтобы мы могли их
исправить.

## GitHub issues – наша система отслеживания задач {#github-issues-is-our-ticketing-platform}

О проблемах следует сообщать на GitHub в виде [GitHub
issues](https://github.com/tuist/tuist/issues), а не в Slack или на других
платформах. GitHub лучше подходит для отслеживания и управления проблемами,
находится ближе к кодовой базе и позволяет нам отслеживать ход их решения. Кроме
того, он поощряет развёрнутое описание проблемы, что заставляет репортера глубже
продумать её и предоставить больше контекста.

## Контекст имеет решающее значение {#context-is-crucial}

Issue без достаточного контекста считается неполным, и автору будет предложено
предоставить дополнительную информацию. Если она не будет предоставлена, issue
будет закрыт. Подумайте об этом так: чем больше контекста вы дадите, тем проще
нам будет понять проблему и исправить её. Поэтому, если вы хотите, чтобы issue
был исправлен, предоставьте как можно больше контекста. Постарайтесь ответить на
следующие вопросы:

- Что вы пытались сделать?
- Как выглядит ваш график?
- Какую версию Tuist вы используете?
- Это вас блокирует?

Мы также требуем предоставить минимальный **воспроизводимый проект**.

## Воспроизводимый проект {#reproducible-project}

### Что такое воспроизводимый проект? {#what-is-a-reproducible-project}

Воспроизводимый проект – это небольшой проект Tuist для демонстрации проблемы.
Часто эта проблема вызвана ошибкой в Tuist. Ваш воспроизводимый проект должен
содержать минимум функций, необходимых для наглядной демонстрации ошибки.

### Зачем нужно создавать воспроизводимые тестовые примеры? {#why-should-you-create-a-reproducible-test-case}

Воспроизводимые проекты позволяют выявить причину проблемы, что является первым
шагом к ее устранению! Самая важная часть любого сообщения об ошибке – это
описание точных шагов, необходимых для воспроизведения ошибки.

Воспроизводимый проект – это отличный способ поделиться информацией о конкретном
окружении, которое вызывает ошибку. Ваш воспроизводимый проект – лучший способ
помочь людям, которые хотят помочь вам.

### Шаги по созданию воспроизводимого проекта {#steps-to-create-a-reproducible-project}

- Создайте новый git-репозиторий.
- Инициализируйте проект с помощью `tuist init` в каталоге репозитория.
- Добавьте код, необходимый для воссоздания возникшей ошибки.
- Опубликуйте код (для этого подойдет ваш аккаунт на GitHub), а затем ссылайтесь
  на него при создании ошибки.

### Преимущества воспроизводимых проектов {#benefits-of-reproducible-projects}

- **Меньшая область анализа:** при удалении всего лишнего ошибку проще
  обнаружить.
- **Не нужно публиковать конфиденциальный код:** вы можете не иметь возможности
  опубликовать основной проект (по разным причинам). Воссоздав небольшую его
  часть в виде воспроизводимого тестового примера, вы сможете публично
  продемонстрировать проблему, не раскрывая никакой секретной информации.
- **Доказательство наличия бага:** иногда ошибка возникает из-за определённой
  комбинации настроек на вашей машине. Воспроизводимый тестовый пример позволяет
  участникам скачать ваш проект, собрать его и протестировать на своих машинах.
  Это помогает подтвердить проблему и сузить круг возможных причин.
- **Помощь в исправлении ошибки:** если кто-то другой сможет воспроизвести вашу
  проблему, у него часто есть хорошие шансы её исправить. Практически невозможно
  устранить ошибку, не имея возможности сначала её воспроизвести.
