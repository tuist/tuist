# 이슈 리포팅 {#issue-reporting}

Tuist 사용자로서 버그나 예상치 못한 동작을 경험할 수 있습니다. 그러한 경우, 문제를 해결할 수 있도록 신고해 주시기 바랍니다.

## GitHub 이슈는 저희 티켓팅 플랫폼입니다 {#github-issues-is-our-ticketing-platform}

이슈는 Slack이나 다른 플랫폼이 아닌 GitHub에서 [GitHub
이슈](https://github.com/tuist/tuist/issues)로 보고해야 합니다. GitHub는 이슈를 추적하고 관리하는 데 더
적합하며 코드베이스에 더 가깝고 이슈의 진행 상황을 추적할 수 있습니다. 또한 문제에 대한 긴 형식의 설명을 장려하여 보고자가 문제에 대해 더
많이 생각하고 더 많은 맥락을 제공하도록 유도합니다.

## 문맥이 매우 중요합니다 {#context-is-crucial}

충분한 맥락이 없는 이슈는 불완전한 것으로 간주되며 작성자에게 추가 맥락을 요청합니다. 컨텍스트가 제공되지 않으면 이슈가 닫힙니다. 더 많은
컨텍스트를 제공할수록 문제를 더 쉽게 이해하고 해결할 수 있습니다. 따라서 문제를 해결하려면 가능한 한 많은 전후 맥락을 제공하세요. 다음
질문에 답해 보세요:

- 무엇을 하려고 했나요?
- 그래프는 어떻게 보이나요?
- 어떤 버전의 Tuist를 사용하고 계신가요?
- 이것이 방해가 되나요?

또한 최소한의 재현 가능한 프로젝트( **)를 제공해 주셔야 합니다.**

## 재현 가능한 프로젝트 {#reproducible-project}

### 재현 가능한 프로젝트란 무엇인가요? {#what-is-a-reproducible-project}

재현 가능한 프로젝트는 문제를 보여주기 위한 작은 Tuist 프로젝트입니다. 이 문제는 대개 Tuist의 버그로 인해 발생합니다. 재현 가능한
프로젝트에는 버그를 명확하게 보여주기 위해 필요한 최소한의 기능만 포함되어야 합니다.

### 재현 가능한 테스트 케이스를 만들어야 하는 이유는 무엇인가요? {#why-should-you-create-a-reproducible-test-case}

재현 가능한 프로젝트는 문제의 원인을 분리할 수 있게 해주며, 이는 문제를 해결하기 위한 첫걸음입니다! 버그 보고서의 가장 중요한 부분은 버그를
재현하는 데 필요한 정확한 단계를 설명하는 것입니다.

재현 가능한 프로젝트는 버그를 유발하는 특정 환경을 공유하는 훌륭한 방법입니다. 여러분의 재현 가능한 프로젝트는 도움을 주고자 하는 사람들에게
가장 효과적인 지원 수단입니다.

### 재현 가능한 프로젝트 생성 단계 {#steps-to-create-a-reproducible-project}

- 새 git 저장소를 생성하세요.
- 저장소 디렉터리에서 ` `` 또는 `tuist init` ` 명령어로 프로젝트를 초기화하십시오.
- 발생한 오류를 재현하는 데 필요한 코드를 추가하세요.
- 코드를 게시하세요(GitHub 계정이 적합한 장소입니다). 이슈 생성 시 해당 코드 링크를 첨부하세요.

### 재현 가능한 프로젝트의 이점 {#benefits-of-reproducible-projects}

- **더 작은 표면적:** 오류만 남기고 나머지를 제거하면 버그를 찾기 위해 깊이 파고들 필요가 없습니다.
- **비밀 코드 공개 불필요:** 여러 이유로 메인 사이트를 공개할 수 없을 수 있습니다. 작은 부분을 재현 가능한 테스트 케이스로 재구성하면
  비밀 코드를 노출하지 않고도 문제를 공개적으로 시연할 수 있습니다.
- **버그 재현 링크:** 때로는 버그가 사용자의 컴퓨터 설정 조합으로 인해 발생하기도 합니다. 재현 가능한 테스트 케이스를 제공하면
  기여자들이 사용자의 빌드를 다운로드하여 자신의 컴퓨터에서도 테스트할 수 있습니다. 이는 문제의 원인을 확인하고 범위를 좁히는 데 도움이
  됩니다.
- **버그 수정 도움 받기:** 다른 사람이 문제를 재현할 수 있다면, 그들도 문제를 해결할 가능성이 높습니다. 버그를 재현하지 못하면
  해결하는 것은 거의 불가능합니다.
