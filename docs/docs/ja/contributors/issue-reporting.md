# 問題報告{#issue-reporting}

Tuistをご利用中、不具合や予期せぬ動作に遭遇する可能性があります。そのような場合は、修正のためぜひご報告ください。

## GitHub issues は当社のチケット管理プラットフォームです{#github-issues-is-our-ticketing-platform}

問題はSlackや他のプラットフォームではなく、[GitHub
issues](https://github.com/tuist/tuist/issues)としてGitHubで報告してください。GitHubは問題の追跡と管理に適しており、コードベースに近く、問題の進捗を追跡できます。さらに、問題の詳細な説明を促すため、報告者が問題を考え、より多くの背景情報を提供することを強制します。

## 文脈が極めて重要です{#context-is-crucial}

十分なコンテキストがない問題は不完全と見なされ、追加のコンテキストが求められます。提供されない場合、問題はクローズされます。こう考えてください：提供されるコンテキストが多ければ多いほど、問題を理解し修正しやすくなります。したがって、問題を修正してほしい場合は、可能な限り多くのコンテキストを提供してください。以下の質問に答えるよう努めてください：

- 何をしようとしていたのですか？
- グラフはどうなっていますか？
- お使いのTuistのバージョンは何ですか？
- これで問題が解決しましたか？

また、最小限の**再現可能なプロジェクトを提供する必要があります。**

## 再現可能なプロジェクト{#reproducible-project}

### 再現可能なプロジェクトとは何ですか？{#what-is-a-reproducible-project}

再現可能なプロジェクトとは、問題を実証するための小さなTuistプロジェクトです。多くの場合、この問題はTuistのバグによって引き起こされます。再現可能なプロジェクトには、バグを明確に示すために必要な最小限の機能のみを含めるべきです。

### なぜ再現可能なテストケースを作成すべきですか？{#why-should-you-create-a-reproducible-test-case}

再現可能なプロジェクトは問題の原因を特定することを可能にし、それが解決への第一歩です！バグ報告で最も重要なのは、そのバグを再現するために必要な正確な手順を記述することです。

再現可能なプロジェクトは、バグを引き起こす特定の環境を共有する優れた方法です。あなたの再現可能なプロジェクトは、あなたを助けたい人々を支援する最良の方法です。

### 再現可能なプロジェクトを作成する手順{#steps-to-create-a-reproducible-project}

- 新しいgitリポジトリを作成する。
- ` リポジトリディレクトリ内で、``` を使用してプロジェクトを初期化してください。
- 確認したエラーを再現するために必要なコードを追加してください。
- コードを公開し（GitHubアカウントが適しています）、イシュー作成時にそのリンクを貼ってください。

### 再現可能なプロジェクトの利点{#benefits-of-reproducible-projects}

- **より小さな表面積:** エラー以外の要素をすべて除去することで、バグを探す手間が省けます。
- **秘密のコードを公開する必要はありません:**
  （様々な理由で）メインサイトを公開できない場合でも、その一部を再現可能なテストケースとして再構築すれば、秘密のコードを一切公開せずに問題を公に実証できます。
- **バグの再現手順:**
  バグは、お使いのマシンの設定の組み合わせによって引き起こされる場合があります。再現可能なテストケースを用意することで、貢献者があなたのビルドを取得し、自身のマシンでもテストできるようになります。これにより、問題の原因を検証し、特定することが容易になります。
- **バグ修正のサポートはこちら:**
  他の方が問題を再現できる場合、修正できる可能性が高くなります。問題を再現できなければ、バグを修正することはほぼ不可能です。
